# 跑步 Android 应用产品与技术规划

## 1. 产品定位

这是一款面向大众跑者的跑步记录与激励应用，形态接近 Nike Run Club。第一阶段的成败不取决于社区功能有多丰富，而取决于用户锁屏跑完一段路后，应用是否能完整、准确、可信地记录这次跑步。

因此，产品优先级应该是：

1. 跑步追踪准确可靠。
2. 跑步过程中体验清晰、安全、不打扰。
3. 历史数据可回看、可分析。
4. 再扩展游戏化、社交、挑战和社区。

## 2. 核心模块

| 模块 | 优先级 | 说明 |
| --- | --- | --- |
| 跑步追踪 Core Tracker | P0 | 户外 GPS、室内跑步、距离、配速、时长、卡路里、暂停/恢复 |
| 数据可靠性 | P0 | 本地持久化、崩溃恢复、前台服务、后台定位 |
| 历史记录 Analytics | P1 | 跑步详情、分段配速、周/月/年统计、跑鞋寿命 |
| 语音教练 | P1 | 每公里播报、时间播报、配速提醒 |
| 游戏化 | P2 | 勋章、等级、连续跑步、目标激励 |
| 云同步 | P2 | 账号、跨设备同步、备份恢复 |
| 社交社区 | P3 | 好友排行、挑战赛、路线分享、动态广场 |

## 3. Android 技术架构

推荐采用 Kotlin + Jetpack 的主流架构：

```text
UI Layer
  Compose 页面、跑步仪表盘、历史详情、地图轨迹

Domain Layer
  RunSessionEngine、距离/配速计算、GPS 过滤、成就规则

Data Layer
  Location Provider、Room、本地缓存、云同步 Repository

System Layer
  Foreground Service、Notification、SensorManager、权限与电池优化引导
```

核心状态机：

```text
IDLE → PREPARING → RUNNING ↔ PAUSED → FINISHED
                    ↓
                 RECOVERING
```

其中 `RECOVERING` 用于处理应用进程重启、系统回收、崩溃后恢复未完成跑步的情况。

## 4. 关键技术选型

| 模块 | 推荐选型 | 原因 |
| --- | --- | --- |
| 语言 | Kotlin | Android 一线开发语言，协程和 Flow 适合实时数据流 |
| UI | Jetpack Compose | 快速构建跑步仪表盘、历史页和状态驱动 UI |
| 架构 | MVVM / Clean Architecture | 方便隔离系统定位、业务计算和页面展示 |
| 定位 | 高德定位 SDK + 高德地图 | 国内场景定位与地图体验更稳 |
| 后台运行 | Foreground Service + Notification | 跑步追踪必须常驻前台服务通知 |
| 本地数据库 | Room | 按点位持久化跑步数据，支持崩溃恢复 |
| 配置存储 | DataStore | 保存用户偏好、播报设置、目标设置 |
| 室内/弱 GPS | SensorManager / Step Counter | 跑步机和 GPS 弱场景下辅助估算 |
| 语音 | Android TextToSpeech | 快速实现公里播报和语音教练 |
| 后台同步 | WorkManager | 只用于跑后同步，不承担实时追踪 |

## 5. GPS 轨迹数据管线

跑步数据不要直接“收到一个点就加距离”。推荐管线如下：

```text
原始定位点
  ↓
时间戳校验
  ↓
精度阈值过滤
  ↓
异常速度过滤
  ↓
静止/暂停判断
  ↓
轨迹平滑
  ↓
距离与配速计算
  ↓
本地持久化
  ↓
UI / 通知栏 / 语音播报
```

建议同时保存两份数据：

- 原始点：用于调试、复盘、算法迭代。
- 处理后点：用于展示轨迹、计算距离和配速。

### GPS 去噪策略

第一版不必直接上复杂算法，可以先组合几类规则：

- 丢弃 `accuracy` 太差的点，例如大于 30～50 米。
- 丢弃时间戳倒退或间隔异常的点。
- 根据人类跑步速度设置上限，例如超过 8～10 m/s 的点需要谨慎处理。
- 对瞬时配速使用滑动窗口，不直接展示单点速度。
- 静止时不累计距离，避免原地 GPS 漂移刷里程。
- 后续再接入 Kalman Filter 做平滑。

## 6. 后台保活与权限策略

跑步追踪必须使用 Foreground Service，并在通知栏持续展示跑步状态。

需要重点处理：

- `FOREGROUND_SERVICE`
- `FOREGROUND_SERVICE_LOCATION`
- `ACCESS_FINE_LOCATION`
- Android 13+ 的 `POST_NOTIFICATIONS`
- 需要真实后台定位时再申请 `ACCESS_BACKGROUND_LOCATION`
- Android 14+ 前台服务类型限制
- 国产 ROM 的电池优化和后台限制引导

注意：如果用户在应用前台点击“开始跑步”，随后由前台服务持续记录，这和“任意后台偷偷定位”不是同一个权限场景。不要一开始就粗暴索要所有权限，应在具体功能需要时分步申请。

## 7. 里程碑计划

### 阶段 0：技术验证，1～2 周

目标：证明最核心技术链路可行。

- 前台服务常驻通知。
- 锁屏连续定位 2 小时。
- 记录点位到本地。
- 画出轨迹线。
- 计算距离、时长、平均配速。
- 崩溃后可恢复未结束跑步。

验收标准：

- 5km 户外跑步距离误差控制在 2%～3%。
- 锁屏期间点位丢失率低于 1%。
- 应用被系统回收后，最多丢失一个采样周期的数据。

### 阶段 1：MVP，4～6 周

目标：完成可真实使用的跑步记录器。

- 开始/暂停/恢复/结束。
- 跑步中仪表盘。
- 地图轨迹展示。
- 前台服务通知栏控制。
- 本地历史记录。
- 跑步详情页。
- 基础自动暂停。
- GPS 过滤参数调优。

### 阶段 2：Beta，4～6 周

目标：让体验接近正式产品。

- 语音播报。
- 分段配速。
- 周/月统计。
- 跑鞋寿命管理。
- 室内跑步模式。
- 更完整的异常处理。
- 适配主流 Android 机型和 ROM。

### 阶段 3：正式版，6～10 周

目标：接入账号、云端和增长功能。

- 账号系统。
- 数据同步。
- 好友关系。
- 排行榜。
- 勋章与等级。
- 挑战赛。
- 路线分享。
- 社区动态。
- 隐私设置与内容审核。

## 8. 第一版建议边界

第一版不要急着做社区。社区和成就是锦上添花，跑步数据不准则是产品根基塌掉。

建议第一版只承诺：

- 用户能开始一次户外跑。
- 锁屏跑完后数据还在。
- 距离和配速可信。
- 能查看历史和轨迹。
- 能从异常中恢复。

等这个闭环跑稳，再逐步扩展语音、勋章、挑战和社区。
